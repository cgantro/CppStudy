# 구체 타입을 숨기는 기법

C++은 정적 타입 언어지만, 런타임에 다양한 타입을 다루기 위해 타입 소거 기법을 제공한다.

---

## `std::function`

호출 가능한 객체(함수 포인터, 람다, 함수자)를 **하나의 타입으로 감싸는** 래퍼다.  
내부적으로 가상 함수 테이블과 유사한 구조를 heap에 동적 할당하여 타입 소거를 구현한다.

```cpp
#include <functional>
#include <iostream>

void run(const std::function<void()>& task) {
    task();
}

int main() {
    auto lambda = [] { std::cout << "lambda\n"; };
    void (*fptr)() = [] { std::cout << "function pointer\n"; };

    run(lambda);  // 람다
    run(fptr);    // 함수 포인터
    // 어떤 callable이든 동일한 인터페이스로 전달 가능
}
```

- **장점** : 호출 가능한 다양한 타입을 하나로 다룸, 시그니처만 같으면 무엇이든 저장 가능
- **단점** : 간접 호출(가상 디스패치), 소형 버퍼 최적화(SBO) 초과 시 동적 할당, 인라인 불가로 인한 성능 비용
- **언제 쓰나** : 콜백 저장, 이벤트 핸들러, 전략 패턴처럼 호출 타입이 런타임에 결정될 때

## `std::variant`

#### [코드](./code/variant.cpp)

**미리 정해진 타입 집합** 중 하나를 안전하게 저장하는 타입-안전 union이다.  
union과 달리 현재 활성 타입을 내부에 기록하며, 잘못된 타입 접근 시 예외를 던진다.

```cpp
#include <variant>
#include <string>
#include <iostream>

using Event = std::variant<int, std::string>;

void handle(const Event& e) {
    std::visit([](const auto& value) {
        std::cout << value << '\n';
    }, e);
}

int main() {
    Event e1 = 42;
    Event e2 = std::string("Hello, Variant!");

    handle(e1);  // 42
    handle(e2);  // Hello, Variant!

    // 안전한 접근
    if (auto* p = std::get_if<int>(&e1)) {
        std::cout << "int: " << *p << '\n';
    }

    // 잘못된 타입 접근 → std::bad_variant_access 예외
    // std::get<std::string>(e1);
}
```

### `std::visit`

`std::visit(visitor, variant)`은 variant에 저장된 **실제 타입에 맞는 오버로드**를 골라 호출한다.  
`auto&` 람다 하나로 모든 타입을 처리하거나, `if constexpr`로 타입별 분기가 가능하다.

```cpp
std::visit([](const auto& v) {
    using T = std::decay_t<decltype(v)>;
    if constexpr (std::is_same_v<T, int>)
        std::cout << "int: " << v << '\n';
    else
        std::cout << "string: " << v << '\n';
}, e1);
```

- **장점** : 힙 할당 없음, 타입 안전, 빠짐없는 타입 처리를 컴파일 타임에 강제
- **단점** : 저장 가능한 타입을 컴파일 타임에 고정해야 함
- **언제 쓰나** : 상태 머신, 이벤트/메시지 처리, 파싱 결과처럼 타입 후보가 명확할 때

## `std::optional`

값이 **있을 수도, 없을 수도** 있는 상황을 표현한다.  
포인터처럼 null 여부를 체크하지만 힙 할당이 없고, 의도가 코드에 명확히 드러난다.

```cpp
#include <optional>
#include <string>
#include <iostream>

std::optional<std::string> find_user(int id) {
    if (id == 1) return "Alice";
    return std::nullopt;  // 값 없음
}

int main() {
    auto result = find_user(1);

    // 방법 1: has_value / value
    if (result.has_value())
        std::cout << result.value() << '\n';  // Alice

    // 방법 2: operator bool + operator*
    if (result)
        std::cout << *result << '\n';

    // 방법 3: value_or — 없으면 기본값
    std::cout << find_user(99).value_or("unknown") << '\n';  // unknown

    // 값 없는 상태에서 value() 호출 → std::bad_optional_access 예외
    // find_user(99).value();
}
```

- **장점** : 힙 할당 없음, null 가능성이 타입에 명시되어 실수 방지, `value_or`로 기본값 처리 간결
- **단점** : 복잡한 부재 이유(에러 종류 등)는 표현 불가 → `std::expected`(C++23) 사용
- **언제 쓰나** : 조회 실패, 설정 미입력, 지연 초기화처럼 "없음"이 정상 케이스인 상황

## 비교 요약

| | `std::function` | `std::variant` | `std::optional` |
|---|---|---|---|
| 목적 | callable 타입 소거 | 여러 타입 중 하나 저장 | 값의 존재 여부 표현 |
| 타입 후보 | 시그니처만 같으면 무제한 | 컴파일 타임에 고정 | T 또는 없음 |
| 힙 할당 | SBO 초과 시 발생 | 없음 | 없음 |
| 잘못된 접근 시 | - | `bad_variant_access` | `bad_optional_access` |
